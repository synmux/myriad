Omi Proactive Notifications

Architecture primer for building a scheduled notification system on Nuxt 4 + Cloudflare Workers

1 How Omi Apps Work

Omi apps are modular extensions that connect to the platform via webhooks. Your app registers capabilities (what it can do) and endpoints (where Omi should send data). There are two key app types relevant here: Integration Apps that receive webhooks when things happen, and Proactive Notification apps that push messages back to users.

Omi App Lifecycle

The crucial thing: you never push directly to the phone. You call Omi's notification API, and Omi relays it to the user's device via FCM. This keeps things simple — you just need an HTTP POST.

2 The Omi Notification API

There are two types of notification you can send.

Direct Notifications

Send an exact text message to a specific user. Good for alerts, reminders, and responses to user actions.

POST /v2/integrations/{app_id}/notification

Params: uid (user ID), message (text)
Auth: Bearer {app_secret}

Proactive Notifications

AI-generated contextual messages based on real-time conversations. You provide a prompt template; Omi generates the personalised message.

Requires capabilities: external_integration + proactive_notification

Rate limit: 1 proactive notification per user per app every 30 seconds. Direct notifications are more flexible but still rate-limited. 
// Sending a direct notification from your Nuxt server route
const response = await fetch(
  `https://api.omi.me/v2/integrations/${OMI_APP_ID}/notification` +
  `?uid=${userId}&message=${encodeURIComponent(message)}`,
  {
    method: 'POST',
    headers: { 'Authorization': `Bearer ${OMI_APP_SECRET}` }
  }
);

3 App Registration

When you create your app in the Omi developer portal (Explore → Create an App), you'll configure it with these capabilities: 

{
  "name": "My Proactive Reminders",
  "capabilities": ["external_integration", "proactive_notification"],
  "external_integration": {
    "triggers_on": "transcript_processed",
    "webhook_url": "https://yourapp.example.com/api/omi/webhook"
  },
  "proactive_notification": {
    "scopes": ["user_context"]
  }
}

The triggers_on: "transcript_processed" setting means Omi will POST to your webhook every time a chunk of real-time transcript is processed. You'll get the transcript segments, a session_id, and the uid of the user. You can also trigger on memory_created for post-conversation summaries.

4 Scheduling Architecture

The core challenge: a user says "remind me to call the dentist in 4 hours" and your system needs to (a) parse that intent, (b) schedule a future action, and (c) fire the notification at the right time. Here's how to architect this on Cloudflare.

Scheduling Options on Cloudflare
Recommendation: Cloudflare Workflows. They're purpose-built for this. step.sleep('4 hours') literally just works — it durably persists, costs nothing while sleeping, and auto-retries on failure. For a "remind me in X" use-case, this is the cleanest approach by far.

7 Workflow Implementation Sketch

// server/workflows/reminder.ts
import { WorkflowEntrypoint, WorkflowStep, WorkflowEvent } from 'cloudflare:workers';

interface ReminderParams {
  userId: string;
  message: string;
  deliverAt: string;  // ISO 8601
  reminderId: string;
}

export class ReminderWorkflow extends WorkflowEntrypoint {
  async run(event: WorkflowEvent<ReminderParams>, step: WorkflowStep) {
    const { userId, message, deliverAt, reminderId } = event.params;

    // Step 1: Validate and persist to D1
    await step.do('persist reminder', async () => {
      await this.env.DB.prepare(
        `INSERT INTO reminders (id, user_id, message, deliver_at, status)
         VALUES (?, ?, ?, ?, 'scheduled')`
      ).bind(reminderId, userId, message, deliverAt).run();
    });

    // Step 2: Sleep until delivery time
    await step.sleepUntil('wait for delivery time', new Date(deliverAt));

    // Step 3: Send the notification via Omi API
    await step.do('send notification', async () => {
      const res = await fetch(
        `https://api.omi.me/v2/integrations/${this.env.OMI_APP_ID}/notification` +
        `?uid=${userId}&message=${encodeURIComponent(message)}`,
        {
          method: 'POST',
          headers: { Authorization: `Bearer ${this.env.OMI_APP_SECRET}` }
        }
      );
      if (!res.ok) throw new Error(`Omi API error: ${res.status}`);
    });

    // Step 4: Mark as delivered
    await step.do('mark delivered', async () => {
      await this.env.DB.prepare(
        `UPDATE reminders SET status = 'delivered' WHERE id = ?`
      ).bind(reminderId).run();
    });
  }
}

8 Handling Cancellation

One powerful feature of Workflows is step.waitForEvent(). You can race a sleep against a cancellation event, so if a user says "cancel my reminder", you send an event to the running workflow instance and it stops. 

// Alternative Step 2: Race sleep vs. cancel
try {
  await step.waitForEvent('await cancel or timeout', {
    event: 'cancel',
    timeout: deliverAt  // timeout = delivery time
  });
  // If we get here, user cancelled
  await step.do('mark cancelled', async () => {
    await this.env.DB.prepare(
      `UPDATE reminders SET status = 'cancelled' WHERE id = ?`
    ).bind(reminderId).run();
  });
  return; // Exit without sending notification
} catch (e) {
  // Timeout reached = time to deliver!
  // Continue to step 3…
}
Cancellation flow: When your webhook handler detects a "cancel my reminder" intent, find the workflow instance ID (stored in D1 alongside the reminder) and send an event to it via the Workflows API: env.REMINDER_WORKFLOW.sendEvent(instanceId, { event: 'cancel' })

9 Webhook Handler Design

// server/api/omi/webhook.post.ts
export default defineEventHandler(async (event) => {
  const body = await readBody(event);
  const { session_id, segments, uid } = body;

  // Accumulate transcript segments in KV for context
  const existing = await event.context.cloudflare.env.KV
    .get(`session:${session_id}`, 'json') ?? [];
  existing.push(...segments);
  await event.context.cloudflare.env.KV
    .put(`session:${session_id}`, JSON.stringify(existing), { expirationTtl: 3600 });

  // Run AI intent extraction on accumulated transcript
  const transcript = existing.map(s => s.text).join(' ');
  const intent = await extractIntent(transcript, event.context.cloudflare.env);

  if (intent?.type === 'reminder') {
    // Spawn a Workflow instance
    const reminderId = crypto.randomUUID();
    const instance = await event.context.cloudflare.env.REMINDER_WORKFLOW
      .create({ id: reminderId, params: {
        userId: uid,
        message: intent.message,
        deliverAt: intent.deliverAt,
        reminderId
      }});

    // Confirm to user via immediate notification
    await sendOmiNotification(uid, `✓ Reminder set: "${intent.message}"`);
  }

  if (intent?.type === 'cancel_reminder') {
    // Look up active workflow instance and cancel
    const reminder = await findActiveReminder(uid, intent.query);
    if (reminder) {
      await event.context.cloudflare.env.REMINDER_WORKFLOW
        .sendEvent(reminder.id, { event: 'cancel' });
    }
  }

  return { status: 'ok' };
});

10 Key Considerations

Deduplication

Transcripts arrive in segments. A user saying "remind me at 3pm" might span 2–3 webhook calls. Use KV to accumulate segments per session and run intent extraction on the full context. Use a deduplication key (e.g. hash of intent + timestamp) to avoid spawning duplicate workflows.

Timezone Handling

"Remind me at 3pm" needs the user's timezone. Store this in D1 as a user preference (set during onboarding or extracted from their Omi profile). Without it, default to UTC and confirm with the user.

Relative vs. Absolute Times

"In 4 hours" is straightforward — compute from Date.now(). "At 3pm tomorrow" requires timezone + date parsing. Use a library like date-fns-tz or handle via your AI extraction prompt (return ISO 8601).

Event-Based Triggers

"Remind me when I get home" is harder — it requires location or context awareness. Consider storing these as conditional reminders in D1 and evaluating them against user context during subsequent transcript webhooks.

Rate Limits

Omi allows 1 proactive notification per user per app every 30 seconds. For scheduled reminders using direct notifications, you have more headroom, but still batch multiple due reminders with a short delay between calls.

Workflow Limits

CF Workflows state persists for 3 days (free) or 30 days (paid). For reminders further out, consider storing in D1 and using a cron trigger to spawn workflows a few hours before delivery time.

D1 Schema Sketch
CREATE TABLE reminders (
  id           TEXT PRIMARY KEY,
  user_id      TEXT NOT NULL,
  message      TEXT NOT NULL,
  deliver_at   TEXT NOT NULL,   -- ISO 8601
  status       TEXT DEFAULT 'scheduled',  -- scheduled | delivered | cancelled | failed
  workflow_id  TEXT,           -- CF Workflow instance ID for cancellation
  created_at   TEXT DEFAULT (datetime('now')),
  delivered_at TEXT
);

CREATE INDEX idx_reminders_user ON reminders(user_id, status);
CREATE INDEX idx_reminders_deliver ON reminders(deliver_at, status);

CREATE TABLE users (
  omi_uid      TEXT PRIMARY KEY,
  timezone     TEXT DEFAULT 'UTC',
  preferences  TEXT,  -- JSON blob
  created_at   TEXT DEFAULT (datetime('now'))
);
